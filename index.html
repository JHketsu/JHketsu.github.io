一日一题：无重复字符的最长子串

发表于 2019-01-22 | 分类于 oj刷题

题目

给定一个字符串，请你找出其中不含有重复字符的 最长子串 的长度
示例

输入: “abcabcbb”
输出: 3
解释: 因为无重复字符的最长子串是 “abc”，所以其长度为 3。
输入: “bbbbb”
输出: 1
解释: 因为无重复字符的最长子串是 “b”，所以其长度为 1。
输入: “pwwkew”
输出: 3
解释: 因为无重复字符的最长子串是 “wke”，所以其长度为 3。
请注意，你的答案必须是 子串 的长度，”pwke” 是一个子序列，不是子串。

我的思路

拿到题目，感觉像是dp或贪心之类的题目，但毫无头绪。查阅文档后，直接利用string的find函数解题，整体思路比较直接，以每个下标对应字母为子串的头部，直到找到尾部，maxlength记录最大子串的长度。

算法

j从左向右遍历字符数组，i记录子串左端，index存储扫描过的字符最后一次出现的位置的下一位，那么当一个子串中出现重复字符时（注意这个字符目前不算做扫描过的字符），i的下标会更新为index位，即：对于abcda，当j扫描到第二个a时，i由原来第一个a的位置移动到其下一位，也就是b的位置。
与二重循环相比，这种方法减少了不必要的子串扫描，比如对于dabca来说，二重循环会从每一位开始找子串，而一重循环的子串会跳过第一个a。
该算法还巧用ASCII码，将每种字符唯一表示。

一日一题：两数之和

发表于 2019-01-20 | 分类于 oj刷题

题目

leetcode-1：两数之和

给定一个整数数组 nums 和一个目标值 target，请你在该数组中找出和为目标值的那 两个 整数，并返回他们的数组下标。
你可以假设每种输入只会对应一个答案。但是，你不能重复利用这个数组中同样的元素。

示例：
给定 nums = [2, 7, 11, 15], target = 9
因为 nums[0] + nums[1] = 2 + 7 = 9
所以返回 [0, 1]

我的思路

代码

时间效率分析

O(n^2)
如果使用二分法，那么是O(n*logn)

别人的代码

我的代码用时72MS
最快的代码用时4MS

算法

首先用一个具有两个元素每个元素的初始值为0的vector。
再用哈希表来存储已经访问过的元素，元素值作为key，其索引作为value。（这和我们需要比较数字大小这点有关）。
对输入的数组只做一次遍历，从后向前，访问过的元素都加入哈希表中，每次检查最新访问的元素是否和哈希表中存在的元素构成满足题意的两数之和。
仔细看循环条件，发现这种循环的方式也没有用过，for第二个项使用i–，实际上是if(i–),能在i=0时自动退出循环，减少了代码量。

数据结构

unordered_map

C++11中，哈希表对应的容器是unordered_map，适用于需要高效率查询的情况。具体可查阅文档。

GoF设计模式

发表于 2019-01-13 | 分类于 学习笔记-软件工程

Pattern

Pattern is the solution to a recurrent problem, not an concrete solution, but an abstract version of it.
four elements

  1. name of the pattern：to describes the pattern, its solutions and consequences in a word or two
  2. the problem:describes when to apply the pattern
  3. the solution:describes the elements that make up the design, their relationships, responsibilities, and collaborations
  4. consequences:the results and trade-offs in applying the pattern

Organizing Design Patterns

by Purpose

  1. Creational Patterns
  2. Structural Patterns
  3. Behavioral Patterns

by Scope

  1. classes
  2. objects

SingleTon 单实例模式

类的多重性：此类可同时存在对象的数目。
实现

  1. 构造函数私有化
  2. 将对象作为类的private static member
  3. 添加public static get方法将对象初始化（首次使用时初始化）

代码

lazy loading

Adapter 适配器模式

将一个类的接口转换成客户希望的另外一个接口。Adapter模式使原本由于接口不兼容而不能一起工作的那些类可以一起工作。
适配器继承或依赖已有的对象，实现想要的目标接口。
应用环境：主要解决在软件系统中，常常要将一些”现存的对象”放到新的环境中，而新环境要求的接口是现对象不能满足的。

代码

用已有的火鸡去实现目标（鸭子）接口

Facade 外观模式

外观模式是为一套接口提供一个统一的接口，使得子系统更易于使用。（对应GRASP中的控制器）
应用场景：需要使用复杂系统中的一个子集，或以特定的方式与系统互动。
Facade与Adapter

  1. 外观模式定义新的接口，适配器模式使用旧的接口
  2. 外观模式包装许多对象，适配器模式包装单一对象
  3. 外观模式常使用单实例模式，因为只需要一个外观对象（GRASP中的控制器模式）

电脑的例子：
电脑整机是 CPU、内存、硬盘的外观。有了外观以后，启动电脑和关闭电脑都简化了。
直接 new 一个电脑。
在 new 电脑的同时把 cpu、内存、硬盘都初始化好并且接好线。
对外暴露方法（启动电脑，关闭电脑）。
启动电脑（按一下电源键）：启动CPU、启动内存、启动硬盘
关闭电脑（按一下电源键）：关闭硬盘、关闭内存、关闭CPU

Observer 观察者模式

观察者模式定义对象之间的一对多依赖关系，当一个对象改变状态时，所有依赖于它的对象都会自动获得通知。
观察者模式又叫做发布-订阅（Publish/Subscribe）模式、模型-视图（Model/View）模式、源-监听器（Source/Listener）模式或从属者（Dependents）模式。
生活中的例子：彩票、教学、报纸、期刊订阅等。

推模式与拉模式

推模式

推模式是当通知消息来之时，把所有相关信息都通过参数的形式“推给”观察者。（前提是已知观察者需要的信息，于是可以传参）
优点：
1.所有信息通过参数传递过来，直接、简单，观察者可以马上进行处理。
2.观察者与被观察者没有一点联系，两者几乎没有耦合。
缺点：
1.所有信息都强迫推给观察者，不管有用与否。
2.如果想添加一个参数，那就需要修改所有观察者的接口函数。

拉模式

当通知消息来之时，通知的函数不带任何相关的信息，而是要观察者主动去被观察的对象那里去“拉”信息。（其实是把被观察者本身传过去）
优点：
1.可以主动去取自己感兴趣的信息。
2.如要添加一个参数，无需修改观察者。
缺点：
观察者与被观察者有一定的联系 。

Strategy 策略模式

策略模式定义了一组算法，并把它们封装起来，它们之间可以互相替换。策略模式让算法得变化独立于使用它的用户。

Factory 工厂模式

problem:当有特殊考虑时，应当由谁来负责创建对象？
solution:创建称为工厂的纯虚构对象来处理这些创建职责。
工厂模式允许写一个方法去实例化不同的对象，也可以实例化今后新开发的类。这样做不需要修改该方法的代码。
定义一个创建对象的接口，让其子类自己决定实例化哪一个工厂类，工厂模式使其创建过程延迟到子类进行。

Abstract Factory 抽象工厂

抽象工厂模式（Abstract Factory Pattern）是围绕一个超级工厂创建其他工厂。该超级工厂又称为其他工厂的工厂。这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。（抽象之上再抽象？）

Command 命令模式

命令模式将一个请求封装成一个对象，从而能以不同的请求来参数化客户对象、把所有请求存入队列或者写入日志，并且支持撤销操作。它允许您将操作的请求者与实际执行操作的对象分离。
命令对象（系统中有多个）的所有客户（remote），把每个命令对象看作是一个‘black box’. 当客户需要请求对象的服务时，只要简单地调用对象的虚拟execute() 方法。

Composition 组合模式

用户忽略组合对象与单个对象的不同，用户将统一地使用组合结构中的所有对象。将对象组合成树形结构以表示”部分-整体”的层次结构。组合模式使得用户对单个对象和组合对象的使用具有一致性。
组合模式实际上是树形结构的表示，这里具体的可以参考这篇文章，讲的很清楚：《Java 设计模式之组合模式及应用场景》

意思是说，一个组合对象，它的子节点可能还是一个组合对象（树枝节点），也可能是单个对象（叶子节点），而树枝节点和叶子节点都是下级节点，我们可以将它们抽象成一类，把它们的方法也抽象成一套，如果具体类有不能实现的方法，可以抛出异常。
